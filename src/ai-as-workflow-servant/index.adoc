= AI as Workflow Servant +
How Deterministic Scripts Tame AI Unpredictability
:doctype: book
:toc: left

_A pattern for repeatable workflows where a deterministic script drives the process — asking only what it cannot determine itself._

== The Problem

Modern software workflows often involve two kinds of unpredictable actors: *humans* and *AI agents*. Both are creative, context-aware, and capable of judgment — but neither is deterministic. Ask a human to name ten files and you get ten different naming conventions. Ask an AI the same question twice and you may get two different answers.

For repeatable, multi-step workflows — like processing survey data into reports, transforming configuration files across environments, or assembling release notes from commit history — this unpredictability is a liability. Steps get skipped, outputs vary, and debugging becomes archaeology.

At the same time, some steps genuinely benefit from creative intelligence: generating human-readable summaries, choosing sensible defaults for ambiguous inputs, or deciding how to handle unexpected edge cases. A purely scripted pipeline would be rigid and require constant manual updates for these tasks.

== The Insight: AI and Humans Are Equally Unpredictable

It is tempting to treat AI as a fundamentally different kind of tool — something magical that "just knows" what to do. But from a workflow perspective, *an AI agent behaves much like an alternative user*: it reads prompts, makes decisions, and produces output. It is often faster and sometimes more consistent than a human, but it is subject to the same fundamental limitation: _you cannot guarantee what it will do._

This suggests a design principle: *treat AI as you would treat a human operator.* Give it clear instructions, validate its output, and never let it drive the overall process. The same script that asks a human "What should this be called?" can ask an AI the same question — and the script does not need to know or care which one answered.

The goal is to *reduce unpredictability to the minimum that is genuinely unavoidable*: the creative decisions that require judgment. Everything else — sequencing, file handling, API calls, validation — is scripted.

== The Pattern: Orchestrator as Master

The core idea is a *strict separation of concerns*:

* A *deterministic orchestrator* (a plain CLI script) controls the entire workflow. It decides what happens, in what order, and validates every result.
* An *AI agent or human* sits on the other end of stdin/stdout, ready to serve the orchestrator's requests — like a clerk filling out forms on demand.

The orchestrator does _not know_ who is answering. It simply asks questions via `input()` and waits for answers on stdin. This makes it testable, predictable, and independent of any specific AI framework — or of AI at all.

== How It Works

[arabic]
. *Trigger:* The user (or AI) starts the orchestrator with the required arguments. In an AI-assisted setup, a _skill_ file maps the user's intent to the correct orchestrator invocation.
. *Deterministic steps:* The orchestrator runs sub-scripts in sequence — transforming data, calling APIs, validating outputs. No creativity involved.
. *Input needed:* When the orchestrator needs a judgment call, it prints a clear request and blocks on `input()`. Whoever is on the other end — human or AI — provides the answer on stdin.
. *Completion:* The orchestrator exits — with code 0 on success, or non-zero on failure. If an AI is driving, it reports the outcome to the user: a summary on success, or error analysis and suggested next steps on failure.

== Advantages

[cols="1,2",options="header",]
|===
|Advantage |Explanation
|*Determinism where it matters* |Step ordering, file naming, API calls, and validation are fully scripted. Same input → same output, every time.
|*Unpredictability only where unavoidable* |Creative decisions (naming, summarizing, choosing between options) are the only points where human or AI judgment enters the process.
|*Testable without AI* |The orchestrator is a plain CLI program. Pipe pre-recorded answers to stdin and you have a fully reproducible test. No AI framework, no API keys, no non-determinism.
|*Transparent flow* |Every prompt and every response passes through the terminal. Nothing happens "magically" behind the scenes.
|*Actor-agnostic* |The orchestrator works with any respondent: a human typing answers, an AI agent, or a test harness replaying recorded responses. No vendor lock-in.
|*Recoverable* |If an answer causes a validation error or runtime failure, the orchestrator can re-ask. The evaluation of "correct" or "incorrect" is itself deterministic — defined by the script, not by judgment.
|*Narrow information boundary* |The respondent only sees what the orchestrator explicitly includes in its `input()` prompts — never the full workflow state, file system, environment variables, or API responses. This inherent need-to-know boundary makes the pattern amenable to privilege separation — see _Security by Architecture_ below.
|===

=== Security by Architecture

The pattern described in this article creates a natural *privilege separation* — not by accident, but by design. Because the orchestrator is a standalone CLI program that communicates with its respondent exclusively via `input()` on stdin, the respondent — whether human or AI — *never needs access to secrets, credentials, or elevated permissions*.

==== The Privilege Boundary

Consider a workflow that deploys to production, reads database credentials, or accesses an internal API with a service account. In a typical AI-assisted approach, the AI agent would need those credentials itself — they would appear in its context window, its logs, or its environment. This is a security liability.

With the orchestrator pattern, the architecture looks different:

[cols="1,1,1",options="header"]
|===
|Layer |Runs as |Knows secrets?
|*AI agent* |Unprivileged user |No — only sees the orchestrator's `input()` prompts and provides answers on stdin
|*Orchestrator* |Privileged user (via `sudo` or setuid wrapper) |Yes — reads credentials from keychain, vault, or protected files
|*Sub-scripts* |Inherit orchestrator's privileges |Yes — use secrets for API calls, DB access, etc.
|===

The AI is confined to the *creative, unprivileged* side of the boundary. Secrets flow within the deterministic, scripted pipeline — never across the stdin/stdout interface where the AI sits.

==== Enforcing the Boundary on Unix Systems

On Linux and macOS, the orchestrator can run with elevated privileges using two mechanisms:

*Option 1: `sudo` with a dedicated rule* (recommended for most cases):

----
# /etc/sudoers.d/workflow
copilot_user ALL=(deploy_svc) NOPASSWD: /opt/workflows/orchestrate.py
----

The AI agent runs as `copilot_user` and invokes the orchestrator via `sudo -u deploy_svc`. The orchestrator runs as `deploy_svc`, which owns the credentials. Simple, auditable, no compilation needed.

*Option 2: setuid wrapper* (for environments where sudo configuration is impractical):

[source,c]
----
// wrapper.c — compiled, owned by deploy_svc, chmod u+s
#include <unistd.h>
int main(int argc, char *argv[]) {
    execv("/opt/workflows/orchestrate.py", argv);
    return 1;
}
----

The wrapper runs as `deploy_svc` regardless of who invokes it. The orchestrator inherits those privileges. The AI agent — answering on stdin — remains unprivileged.

NOTE: The setuid bit is ignored for interpreted scripts (shebang files) on both Linux and macOS. A compiled wrapper is required when using this approach.

==== Why Existing Principles Already Support This

Several principles from this pattern directly enable the security boundary:

* *"No actor awareness in the orchestrator"* — The orchestrator does not know that an AI is answering. It never sends secrets to stdin because it never needs to. The AI cannot request secrets because it can only _respond_ to prompts — not initiate actions.
* *"Never exit temporarily"* — The elevated process runs once, from start to finish. There is no window where the AI could intercept or restart the process with different parameters.
* *"Validate everything"* — Even if the AI's answer were malicious (e.g., attempting path traversal in a filename), the orchestrator validates before acting. The deterministic validation code is part of the privileged, auditable script.
* *"Minimize unpredictability"* — Everything that touches secrets is in the deterministic part of the pipeline. The AI is only consulted for judgment calls that do not require credentials.

The result: *the same architecture that makes workflows reproducible and testable also makes them secure by default.*

== Disadvantages & Trade-offs

[cols="1,2",options="header",]
|===
|Disadvantage |Mitigation
|*More upfront engineering* |Writing a proper orchestrator is more work than "just let the AI figure it out". Pays off when the workflow runs repeatedly, or when a single run iterates over many items that must be handled identically. Further mitigated by AI-assisted workflow generation — see _From Pattern to Practice_ below.
|*Rigid step ordering* |The orchestrator defines a fixed pipeline. If the workflow fundamentally changes, the script must be updated. Partially mitigated by making steps idempotent and skippable.
|*Prompt quality matters* |The orchestrator's `input()` prompts must be *self-contained* — each prompt must provide enough context for any respondent to answer without prior knowledge of the workflow state. This is not merely a nice-to-have; it is the *enabling constraint* that makes actor-agnosticism work. A prompt that assumes the respondent has been following along will fail for a fresh AI session or a human operator who just joined. A prompt that includes all relevant context works for anyone. Good prompt design benefits human and AI respondents equally.
|*Latency* |Each interaction adds a round-trip. For workflows with many creative steps, this can be slower than a single "do everything" prompt. Mitigated by caching strategies — the orchestrator can skip `input()` entirely for previously answered prompts with identical call arguments.
|===

== When to Use This Pattern

* *Repeatable workflows* that run more than once (data pipelines, report generation, environment provisioning), or single runs that iterate over many items with identical logic (generating 200 invoices, provisioning 20 environments)
* *Mixed automation* where some steps are deterministic and others require judgment
* *Audit requirements* where you need a traceable record of exactly what happened and what was decided
* *Multi-actor pipelines* that combine scripts, APIs, and human or AI input

== When NOT to Use It

* *One-off tasks without repetitive steps* — if there is no loop and no need for consistency across items, just ask directly
* *Purely creative work* — writing prose, brainstorming, design exploration
* *Highly interactive workflows* — where every step depends on the previous response and there is no deterministic spine to script

== Key Principles

[arabic]
. *The orchestrator holds the scepter.* Human and AI are respondents who only act when asked.
. *No actor awareness in the orchestrator.* It does not know or care whether a human or AI is answering. All communication is via `input()` and `print()`.
. *Minimize unpredictability.* Script everything that can be scripted. Only delegate to a respondent what genuinely requires judgment.
. *Never exit temporarily.* The orchestrator runs from start to finish as a single process. If it needs information, it asks and waits. If a response is invalid, the orchestrator re-prompts — it does not abort and expect to be restarted with different parameters. This counters a common anti-pattern in AI-generated scripts: exiting on the first unexpected input, assuming the caller (the AI) will retry with corrected arguments. A well-designed interactive program recovers by asking again, not by crashing.
. *Validate everything.* After receiving an answer, the orchestrator checks it before proceeding. Trust no one — not the human, not the AI.

== Related Work & Positioning

The idea of combining deterministic code with AI capabilities is not new. Several influential publications and frameworks address overlapping concerns — but from different angles.

=== Anthropic: "Building Effective Agents" (December 2024)

Anthropic's widely-cited blog post footnote:[Anthropic, _Building effective agents_, December 2024. https://www.anthropic.com/research/building-effective-agents] draws a clear architectural distinction between *workflows* ("systems where LLMs and tools are orchestrated through predefined code paths") and *agents* ("systems where LLMs dynamically direct their own processes and tool usage"). The post catalogs five workflow patterns — prompt chaining, routing, parallelization, orchestrator-workers, and evaluator-optimizer — all of which assume that the orchestrating code _knows it is calling an LLM_ and uses model-specific APIs.

The pattern described in this article aligns most closely with Anthropic's "workflow" category, but differs in one fundamental respect: *the orchestrator has no knowledge of AI at all.* It communicates via `input()` and `print()` — not via LLM APIs, tool-use protocols, or agent frameworks. This makes it testable with plain stdin, framework-free, and usable without AI entirely. Where Anthropic's workflows are LLM-centric (the code is designed _around_ the model), this pattern is actor-agnostic (the code is designed around the _task_).

Anthropic's advice to "start with the simplest solution possible" and avoid unnecessary complexity resonates strongly with this pattern's philosophy.

=== LangChain, LangGraph, and Similar Frameworks

Frameworks like LangChain and LangGraph provide abstractions for composing LLM calls into workflows — chains, state machines, routing logic, and tool integration. They are powerful for building LLM-powered applications, but they introduce _framework dependency_ and assume the LLM is a first-class component wired into the code.

This pattern requires no SDK, no API client, and no framework. The orchestrator is a plain CLI program that could have been written in 2005. The AI (if present) sits on the other side of stdin — not inside the code. This means zero vendor lock-in, no dependency on model availability, and trivial testability.

The trade-off is flexibility: LangChain-style frameworks offer richer integration with model capabilities (tool use, structured output, streaming). This pattern deliberately sacrifices that richness for simplicity and independence.

=== Multi-Agent Frameworks (CrewAI, AutoGen)

Multi-agent frameworks like CrewAI and Microsoft's AutoGen let multiple AI agents collaborate — each with a role, tools, and a shared communication protocol. The agents _are_ the orchestrators: they decide what to do, delegate to each other, and synthesize results.

This pattern takes the opposite approach. There is exactly *one* orchestrator, and it is deterministic code — not an AI agent. AI is not a peer; it is a respondent who answers when asked. This is a deliberate architectural choice for workflows where predictability, auditability, and reproducibility outweigh the need for dynamic adaptation.

=== Unix Philosophy and the `llm` CLI

Simon Willison's `llm` command-line tool footnote:[Simon Willison, _llm_ CLI tool. https://llm.datasette.io/] brings LLM capabilities into the Unix pipeline: pipe text in, get AI-generated text out. It embodies the Unix philosophy of small, composable tools — and shares this pattern's preference for stdin/stdout as the integration boundary.

The difference is scope: `llm` is a single-invocation tool (one prompt, one response). This pattern describes _multi-step interactive orchestration_ where the script maintains state across many prompts, validates intermediate results, and controls the overall flow. The `llm` tool could serve as one building block _within_ an orchestrator, but it does not address the orchestration problem itself.

=== What This Pattern Adds

The contribution of this pattern is not a new technology but a *specific architectural stance*: the orchestrator must be a plain, actor-agnostic CLI program that communicates exclusively via stdin/stdout, with no knowledge of AI. This stance has consequences that existing approaches do not fully address:

* *Complete actor-agnosticism* — not just "you can swap models" (as in LangChain), but "the orchestrator does not know AI exists."
* *Testability without infrastructure* — pipe recorded answers to stdin. No API keys, no model endpoints, no framework setup.
* *Natural privilege separation* — the stdin/stdout boundary becomes a security boundary (see _Security by Architecture_ above).
* *Bidirectional applicability* — the same orchestrator works for human operators, AI agents, and automated test harnesses without modification.

== From Pattern to Practice

The orchestrator pattern is powerful, but writing one from scratch requires programming skills. This section explores how the barrier to entry can be lowered — allowing users who are not developers to create, review, and trust orchestrated workflows.

=== Self-Generating Workflows

The pattern described so far assumes that a developer writes the orchestrator. But the orchestrator itself can be *generated by the AI* — under the user's supervision.

==== The Idea

A user describes a desired workflow in natural language: "I need a script that provisions a new customer environment — creates a database, generates API keys, configures DNS, and sends a welcome email." The AI then generates the complete stack:

[arabic]
. The *orchestrator script* — the deterministic spine that drives the workflow
. The *sub-scripts* — for each deterministic step (database creation, DNS configuration, etc.)
. A *setuid wrapper or sudo rule* — so the orchestrator can access credentials without exposing them to the AI
. A *Gherkin test suite* — a human-readable specification that the user can review

The AI compiles the wrapper, runs the tests, and presents the result. The user only needs to _read and approve_ — not write code.

==== Gherkin as Reviewable Specification

This is where the approach becomes powerful for non-developers. Raw Python or Shell code is difficult to review, especially when it runs with elevated privileges. But Gherkin (used by Cucumber, Behave, and similar frameworks) is natural language with minimal structure:

[source,gherkin]
----
Feature: Customer provisioning workflow handles secrets securely

  Scenario: Orchestrator reads database password from vault
    Given the orchestrator runs as user "deploy_svc"
    And the vault contains credentials for "customer-db"
    When the orchestrator reaches the database provisioning step
    Then the password is read from the vault
    And the password is never written to stdout
    And the password is never visible to the AI agent

  Scenario: AI agent cannot access secrets directly
    Given the AI agent runs as user "copilot_user"
    When the AI agent attempts to read "/secure/vault.yml"
    Then access is denied with a permission error

  Scenario: Workflow creates database with correct parameters
    Given a new customer "Acme Corp" with plan "enterprise"
    When the provisioning workflow completes
    Then a database "acme_corp" exists
    And the database user has only INSERT, SELECT, UPDATE permissions
    And no DELETE or DROP permissions are granted
----

The user reads this and confirms: "Yes, that is what I want." The AI generates the step definitions (the executable glue code) behind the scenes. The user does not need to understand the step definitions — only the Gherkin scenarios.

==== The Three-Level Trust Chain

This creates a layered review process where the user can stop at whichever level matches their expertise:

[cols="1,2,1",options="header"]
|===
|Level |What the user reviews |Difficulty
|*1. Gherkin specs* |"Does the workflow do what I described?" |Easy — natural language
|*2. Test results* |"Do all tests pass?" |Trivial — green/red
|*3. Source code* |"Is the implementation correct?" |Hard — but optional if level 1 and 2 are thorough
|===

For most users, levels 1 and 2 are sufficient. Level 3 is for developers or security auditors who want full assurance.

==== The Systematic Bias Risk

There is an important caveat: *the AI generates both the code and the tests.* This is analogous to a developer who writes their own unit tests — if the developer misunderstands the requirement, the tests may encode the same misunderstanding, and both will be internally consistent but wrong.

For example, if the user says "the password must never be logged" and the AI interprets "logged" as "written to a log file" but not "printed to stderr", then both the code and the Gherkin scenario will consistently miss the stderr case. The tests pass. The spec looks correct. But the actual behavior has a gap.

This is not a theoretical concern. It is the *same risk that exists whenever one person writes both implementation and tests* — familiar from software engineering, and solvable with the same mitigation: *independent review*.

==== Mitigation: Independent Verification

The mitigation is straightforward — have the specification reviewed by a party that was *not involved in generating the code*:

*Level 1 — New session (sufficient for most cases):*

Start a fresh AI session with no shared context from the code generation session. Provide only the user's original requirement and the generated Gherkin specs. Ask: "Do these scenarios fully cover the stated requirement? What is missing?"

A new session works because systematic errors arise primarily from *shared conversational context* — the AI made an interpretation early in the code generation session that then influenced everything downstream. A fresh session interprets the Gherkin specs independently, without that context.

*Level 2 — Different model (for high-assurance scenarios):*

Use a different AI model for the review. This adds *architectural diversity* — different models have different training biases and different blind spots. If both models agree the specs are complete, confidence increases significantly.

This level is appropriate when:

* The workflow has access to production systems or sensitive data
* Regulatory requirements demand separation of concerns (analogous to the four-eyes principle)
* The organization's risk profile warrants the additional effort

*Level 3 — Human expert review (for security-critical workflows):*

For workflows that manage credentials, handle financial transactions, or operate critical infrastructure, a human security review of the Gherkin specs (and optionally the code) remains the gold standard. The Gherkin format makes this review significantly easier than reviewing raw code.

[cols="1,2,2",options="header"]
|===
|Level |Reviewer |When to use
|*New session* |Same model, fresh context |Default for all generated workflows
|*Different model* |Different AI model |Production access, sensitive data, regulatory requirements
|*Human expert* |Security engineer or domain expert |Critical infrastructure, financial systems, compliance
|===

These levels are *cumulative* — for a high-assurance workflow, you might use all three. For a routine data-processing pipeline, level 1 is typically sufficient.
